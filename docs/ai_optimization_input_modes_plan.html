AI Optimization Input Modes Specification

This document defines the implemented three-mode input pipeline for modified-mode AI optimization. The purpose is to select the best initial learning-rate preset from scene and flight signals, while keeping feature extraction fast and reliable from uploaded images.

Chapter 1 - Purpose and Scope

This specification defines an input-driven initialization strategy for modified-mode AI optimization with two practical targets: first, reduce the time required to match or beat the loss reached by the baseline run; second, improve final reconstruction quality whenever possible after that point. In this document, baseline explicitly means the normal gsplat run without modified-mode optimization. Evaluation is therefore performed on both time and quality, while feature extraction remains lightweight so most compute stays focused on training.

Evaluation Method

The evaluation procedure is designed to verify both target factors, time and quality, in a repeatable way.

  1. Run a baseline reference first using normal gsplat with no modified-mode optimization.
  2. Record baseline metrics: time-to-target-loss, loss at fixed checkpoints (for example 1k, 3k, 5k), and final loss at the selected stop step.
  3. Run modified mode 1 (exif_only) in a batch sequence (for example 10 runs) so the model can keep learning and propose different initial learning-rate presets across runs.
  4. Keep comparison conditions fixed across baseline and modified runs: same project data, same max steps, same compute environment, and comparable seed policy.
  5. For each run, record both anchors: best-loss anchor (first eval at or after t_best) and end-step anchor.
  6. Compute quality score and time score at both anchors, then compute the run decision score using the larger anchor score.
  7. Compare baseline against intermediate checkpoints (for example run 5 and run 6) and the later checkpoint (run 10) using the same scoring policy.
  8. Mark the approach successful when time-to-baseline-quality is reduced; treat additional quality improvement beyond baseline as an additional gain.
Suggested reporting format: baseline vs run 5 vs run 6 vs run 10, with columns for time-to-target-loss, best-loss step, best-anchored quality, end-step quality, and final run decision score. For detailed equations and operator-level decision logic, see Chapter 4B.
Chapter 2 - The Three Modes

Modified mode uses three stacked input modes in config.

Mode Input Policy Use Case
exif_only Use only direct EXIF and file metadata from the uploaded images. Fastest AI-only mode. Suitable for minimal processing and no scene inference.
exif_plus_flight_plan Use EXIF plus cheap sequence-derived flight-plan features from the same images. Best balance between accuracy and cost when the image set already contains GPS/time/orientation data.
exif_plus_flight_plan_plus_external Use EXIF + flight-plan features + extra cheap image features like vegetation proxy and roughness proxy. Best for richer scene understanding without requiring manual external inputs.
The three modes are stacked. Mode 2 adds on top of Mode 1. Mode 3 adds on top of Mode 2.
Chapter 3 - What Goes Into x

x is the feature vector built from the uploaded image set. The key rule is that every feature should be easy to read from the uploaded images or easy to derive quickly from them. No manual input.

Fallback rule used in all three modes: if a value is missing, fill it and set a missing flag. "Dataset middle value" means the middle value seen in training data for that field.

3.1 Direct EXIF features

These are the cheapest and most reliable inputs because they come straight from image metadata.

Feature Why it helps Example Formula Fallback when missing
Camera make/model Different sensors and optics often converge differently. DJI, Zenmuse X7 camera_make = EXIF.Make, camera_model = EXIF.Model unknown_make and unknown_model; camera_meta_missing=1
Lens model Lens behavior affects sharpness, distortion, and view geometry. 24mm, 28mm equivalent lens_model = EXIF.LensModel unknown_lens; lens_missing=1
Focal length Changes field of view and overlap sensitivity. 8.8mm focal_length_mm = median(EXIF.FocalLength) Dataset middle focal length; focal_missing=1
Aperture, shutter speed, ISO Proxy for exposure quality and motion blur risk. f/2.8, 1/500, ISO 100 aperture_f = median(FNumber), shutter_s = median(ExposureTime), iso = median(ISO) Defaults: f/2.8, 1/250, ISO 100; aperture_missing=1, shutter_missing=1, iso_missing=1
Camera angle (if available) Very important for reconstruction behavior: nadir and oblique runs behave differently. Gimbal pitch near -90 (nadir) or near -70 (oblique) bucket(angle): nadir if a<= -80, oblique if -80<a<= -60, high_oblique otherwise unknown_angle_bucket; angle_missing=1
GPS lat/lon/alt Helps estimate flight path and scene coverage. Not mandatory, but very useful. Project-wide coordinate spread gps_lat_mean = median(lat), gps_lon_mean = median(lon), gps_alt_mean = median(alt) Set GPS-derived values to neutral defaults and set gps_missing=1
Timestamp sequence The order in which photos were captured. Helps estimate path direction and speed changes. Photo 001 -> 002 -> 003 by capture time timestamp_mode = exif if DateTimeOriginal exists else filename_order Use filename order; timestamp_missing=1
Image size / orientation Useful for load estimation and capture consistency. 4000x3000 img_width_median = median(width), img_height_median = median(height) Dataset middle width/height; img_size_missing=1

3.2 Flight-plan features derived from the image set

Flight plan is usually not a single EXIF field. It is inferred cheaply from the image set by combining GPS, time, and heading/orientation patterns across images.

Derived feature How to derive it cheaply What it tells the model Formula Fallback when missing
Flight type Classify GPS path pattern as grid, corridor, orbit, oblique, or mixed. Scene geometry and motion style. rule(path_shape, heading_consistency, angle_mean) -> {grid,corridor,orbit,oblique,mixed} unknown_flight_type; flight_type_missing=1
Camera angle profile Use per-image camera angle stats (mean and spread), from metadata if present. Separates nadir-heavy missions from oblique-heavy missions. camera_angle_profile = (mean(angle), std(angle)) Use unknown-angle profile; angle_profile_missing=1
Average altitude Average EXIF GPS altitude when present. Scale of the reconstruction problem. average_altitude = mean(gps_alt) Dataset middle altitude; altitude_missing=1
Heading consistency Measure how stable the movement direction is across image order. Whether the flight is a straight mission or a turning mission. heading_consistency = clamp(1 - std(turn_delta_deg)/90, 0, 1) 0.5 neutral; heading_missing=1
Coverage spread Compute GPS bounding box and track spread. How broad and diverse the sampled scene is. coverage_spread = clamp(bbox_diag_m/600, 0, 1) 0.0; coverage_missing=1
Overlap proxy Use distance between successive captures relative to estimated footprint. How dense the image sampling is. overlap_proxy = clamp(1 - avg_step_m / max(10, 1.2*avg_alt_m), 0, 1) 0.5 neutral; overlap_missing=1

3.3 External features derived from the images themselves

These are still internal to the dataset. They are not manual inputs; they are cheap summaries computed from the images after upload.

Feature Why it matters Cheap derivation idea Formula Fallback when missing
Vegetation cover percentage Vegetation scenes often converge differently from built scenes. Green-area proxy from RGB images. green_cover = green_hits / pixels, green if (g>1.05r and g>1.05b) 0.0; green_area_missing=1
Vegetation complexity score Dense and irregular vegetation is harder than simple flat green regions. Combine green texture entropy and edge density into one score. veg_complexity = clamp(0.55*green_cover + 0.45*texture_density, 0, 1) 0.5 neutral; veg_complexity_missing=1
Terrain roughness / slope proxy Rough terrain often needs more conservative geometry updates. Use image overlap geometry and local visual variance. roughness_proxy = clamp(std(luma)/64, 0, 1) 0.5 neutral; roughness_missing=1
Texture density Rich texture usually supports faster pose/geometry learning. Edge density or frequency-energy proxy. texture_density = clamp(sum(|gx|+|gy|)/((w-2)(h-2)*255), 0, 1) 0.5 neutral; texture_missing=1
Blur / motion risk Blurred data often needs safer initial settings. Sharpness score from a fast Laplacian measure. blur_risk = 1 - clamp(sum(|laplacian|)/((w-2)(h-2)*255), 0, 1) 0.5 neutral; blur_missing=1
External vegetation input is defined as green-area proxy from RGB images.
Chapter 4 - What y and yhat Mean

The model learns which initial learning-rate preset gives the best result after a short trial, such as 5k steps.

Term Meaning Example
x Feature vector from EXIF and cheap derived scene summaries. Camera model, focal length, flight type, vegetation cover
y Ground-truth best preset after running all candidate presets for 5k steps. balanced or geometry_fast
yhat Model prediction for which preset should win. Highest score assigned to balanced

A simple setup is to define a small preset set and let the model score those presets. That keeps training stable and makes evaluation easy.

Preset Typical behavior
conservative Lower initial LR, safer when scene quality is uncertain or blur/vegetation is high.
balanced Middle ground for mixed scenes and general-purpose flights.
geometry_fast Higher geometry-related LR, useful for clean structured scenes and corridor/grid missions.
appearance_fast More aggressive appearance learning, useful when color/detail fitting is the bottleneck.
If a later version requires a continuous target, y can become a regression target over LR multipliers. For the first version, classification over presets is simpler and cheaper.
Chapter 4B - Learning Loop After yhat

This section defines how the AI decides after yhat is produced. The flow is fixed and repeatable. It compares quality and time against the normal gsplat baseline.

  1. Step 1. Predict and pick one preset. The model outputs scores over all preset candidates and selects the highest score.
    k* = argmaxk(sk), with yhat = [s1, s2, ..., sK](4B.1)
    Meaning: argmax means "pick the index of the largest value". yhat is the model score vector, s_k is the score of preset k, and k* is the selected preset index. Example: if yhat = [0.12, 0.41, 0.33, 0.14], then k* = 2 because 0.41 is the largest score.
  2. Step 2. Run training and find the best-loss step. Training runs with preset k^*. The best-loss step is the step with minimum training loss.
    tbest = argmint L(t)(4B.2)
    Meaning: argmin means "pick the index of the smallest value". L(t) is training loss at step t. t_best is the step where loss is lowest. Example: if loss at steps [1000, 1200, 1400, 1600] is [0.081, 0.073, 0.069, 0.071], then t_best = 1400.
  3. Step 3. Select the best-anchored evaluation checkpoint. The checkpoint is the first eval step at or after t_best. If that does not exist, the last eval step is used.
    teval,best = min { t ∈ E | t ≥ tbest }; else teval,best = max(E)(4B.3)
    Meaning: E is the set of eval steps. min{...} means "the smallest value that satisfies the condition". max(E) means "the largest eval step". Example: if E = [1000, 2000, 3000] and t_best = 1400, then t_eval,best = 2000.
  4. Step 4. Compute quality score and time score. Quality uses normalized PSNR, SSIM, and LPIPS. Time uses normalized runtime to a reference.
    Q(t) = wpsnr PSNRnorm(t) + wssim SSIMnorm(t) + wlpips (1 - LPIPSnorm(t))(4B.4)
    T(t) = 1 - clamp( time_to_step(t) / timeref, 0, 1 )(4B.5)
    Meaning: Q(t) is quality score and T(t) is time score. clamp(x,0,1) means force x into the range [0,1]. LPIPS is inverted by (1 - LPIPS_norm) so larger is better, like PSNR and SSIM. Example: if time_to_step(t)/time_ref = 1.2, then clamp = 1 and T(t) = 0. If ratio is 0.6, then T(t) = 0.4.
  5. Step 5. Compute composite score at both anchors. The score at any checkpoint combines quality and time, with higher weight on quality.
    S(t) = wq Q(t) + wt T(t), where wq + wt = 1 and wq > wt(4B.6)
    Meaning: S(t) is one final score at step t. w_q is quality weight and w_t is time weight. Example: with w_q=0.7, w_t=0.3, Q=0.80, T=0.40, then S=0.7*0.80 + 0.3*0.40 = 0.68.
  6. Step 6. Decide run score from best-anchor and end-step. One score is computed at t_eval,best. Another score is computed at t_end. The larger one is used.
    Srun = max( S(teval,best), S(tend) )(4B.7)
    Meaning: max(a,b) means choose the larger value. S_run is the final run decision score and t_end is the final training step. Example: if S(t_eval,best)=0.68 and S(t_end)=0.63, then S_run=0.68.
  7. Step 7. Write transition and update the model. Baseline is always the normal gsplat run without modified mode. The transition (x, yhat, k^*, outcomes, S_{run}) is saved, and model weights are updated from accumulated transitions. Example: one stored row can be (x_project_021, [0.12,0.41,0.33,0.14], 2, outcomes_run10, 0.68).
yhat is the decision signal. Observed run outcomes and baseline-relative scores are the learning signal.

Chapter 6 - Implementation Summary
  1. Use the three explicit stacked modes in config.
  2. Keep the feature extractor cheap and deterministic.
  3. Cache per-project summaries so the input vector is built once, not repeatedly.
  4. Use a small preset set first, then train the selector to pick the best preset after the 5k-step trial.
  5. Use a data strategy with a 10-project cap and balanced coverage across camera profile, flight geometry, scene content, and capture conditions.
  6. Apply 7 train projects and 3 final test projects as the primary split for this phase.
  7. Only add more features if they improve test behavior on unseen scenes.

This provides a low-cost system that can still learn which kind of scene prefers which initial learning rate pattern.

Chapter 7 - Implementation Constraints
  • EXIF alone can give camera and exposure data, but not a full flight plan.
  • Flight plan must be inferred from the sequence of images, usually using GPS and timestamps.
  • External vegetation input is green-area proxy from RGB imagery.
  • Use only cheap preprocessing in the first version so feature extraction does not become the bottleneck.
  • Camera angle should be included with flight type; flight type alone is not enough.
Mode stack: exif_only -> exif_plus_flight_plan -> exif_plus_flight_plan_plus_external.
Chapter 8 - Data Strategy

Data design for this phase prioritizes diversity over volume. With a capped dataset size, coverage across scene and flight conditions is more important than adding near-duplicate projects.

Dataset Size (Current Phase)

  • Maximum in this phase: 10 projects.
  • Minimum workable set: 8 projects.
  • Target set for this phase: 10 projects with strong variety.

Required Variety Groups

Group What to vary Target coverage Why this matters
Group A - Camera Profile Different camera models, focal lengths, and exposure ranges. At least 2 camera families. Initial LR behavior can change with optics and sensor response.
Group B - Flight Geometry Grid, corridor, mixed paths, plus angle mix (nadir and oblique). At least 4 projects with strong oblique coverage. Path shape and angle directly affect convergence dynamics.
Group C - Scene Content Built-up areas, dense vegetation, sparse vegetation, and mixed scenes. At least 2 built-up, 2 vegetation, 2 mixed scenes. Scene complexity shifts the best preset and learning speed.
Group D - Capture Conditions Stable lighting, high contrast, mild blur, and challenging weather/light. At least 2 challenging-condition projects. Robustness requires exposure to non-ideal capture conditions.

Evaluation Split (Primary)

  • Primary split: 7 train projects + 3 final test projects.
  • Project-level validation split is optional in this phase when in-run validation metrics are already used during training.
  • If explicit holdout validation is required, use 6 train + 1 validation + 3 final test.

Continual Learning With Limited Data

One persistent model is used and updated across runs. The model can keep learning even with a small number of projects. Re-running the same project multiple times is valid and helps the model learn run-level policy behavior.

  • Repeated runs on one project provide valid policy-learning signal and can improve project-specific convergence behavior.
  • Project-specific gains do not guarantee cross-project performance; multi-project diversity is required for generalization.
  • Heavy reuse of one project increases overfitting risk, so repeated same-project runs should remain in development and be excluded from final test evidence.
With a 10-project cap, prioritize diversity of camera profile, flight geometry, and scene type over project count.
